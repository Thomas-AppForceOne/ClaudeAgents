# U1 — Project overlay UX

## Problem

C3 defines the project overlay's *schema* — what splice points exist, how `discardInherited` works, how schemaVersion is enforced. That makes the file machine-validatable but does not address what the experience feels like for the human authoring it.

A user creating `.claude/gan/project.md` for the first time needs:

- A discoverable starting point (where the file goes, what to name it, what's the minimum content).
- Clear feedback when their edits are wrong (validation errors that point at the right field).
- An understandable mental model of how project values combine with user-level defaults.
- A way to inspect the resolved view to confirm changes took effect.

This spec covers the human-side concerns that sit on top of C3's machine schema.

## Proposed change

### Bootstrapping

A user can author `.claude/gan/project.md` from scratch in their editor. The minimum valid file is:

```
---
schemaVersion: 1
---
```yaml
{}
```
```

This is an empty overlay — every value falls through to the user-level overlay or defaults. Useful as a placeholder when a project knows it will eventually customise something.

A more typical starter, generated by `gan config set runner.thresholdOverride 8` (R3) when the file does not yet exist:

```
---
schemaVersion: 1
---
```yaml
runner:
  thresholdOverride: 8
```
```

`.claude/gan/project.md` is committed to the repo (zone 1 per F1). It is intended for review like any other config artifact.

### Validation feedback

When `validateAll()` (F2) fails on a project overlay, the user sees a structured report. The report's content surface for the human is:

```
.claude/gan/project.md: 2 issues

  line 8:  Unknown splice point `proposer.weight`. Did you mean `proposer.additionalCriteria`?
  line 14: `runner.thresholdOverride` must be an integer (got "8" as a string).

Run `gan validate` to re-check after editing.
```

Validation never silently ignores a misspelled key — every unknown splice point produces an error with a similar-name suggestion when applicable.

### Mental model documentation

The repo's README ships a one-page conceptual guide:

- The cascade: default → user → project (visual diagram).
- "When you set a value here, it merges with what your user overlay said. When you want to ignore upstream values entirely, use `discardInherited`."
- Examples of each splice point with a one-line use case.
- Pointer to `gan config print` for inspecting the resolved view.

This guide lives alongside the README, not in the spec set, but is part of the U1 deliverable.

### Inspecting the result

After editing, the user runs `gan config print` (R3) to see what the agents will actually receive. The output is the same shape as F2's `getResolvedConfig()` JSON, with formatting that highlights:

- Which splice points came from which tier.
- Which fields had `discardInherited` applied (per O1's `discarded` array).
- Which `additionalContext` files resolved or were missing.

### When to use overlay vs. project-tier stack file

Cross-link to C5: project-tier stack files (`.claude/gan/stacks/<name>.md`) replace stack contents wholesale; overlays add splice-point values. Pick the smaller hammer first. C5's spec carries the detailed comparison; U1 just points users at it.

### Examples

A small example library lives at `examples/project-overlay/` showing:

- A minimal threshold override.
- Adding a project-specific `additionalCriteria` entry.
- Forcing a stack via `stack.override` with `discardInherited`.
- Adding `additionalContext` paths.

These are real, validatable files referenced by the README's mental-model guide.

## Acceptance criteria

- A first-time user runs `gan config set runner.thresholdOverride 8`; the resulting `.claude/gan/project.md` validates cleanly and the value appears in `gan config print` output.
- Editing the same file by hand to introduce a typo produces a `gan validate` error citing the line and field with a similar-name suggestion when applicable.
- The README's overlay guide describes the cascade visually and links to every splice point with a one-line use case.
- `examples/project-overlay/*` files all validate against `schemas/overlay-vN.json`.
- The user-facing error and example surface follows the roadmap's user-facing-discipline rule (no maintainer-only script names).

## Dependencies

- C3 (overlay schema this builds UX on)
- C4 (cascade semantics the mental model describes)
- F2, R1 (validation pipeline producing the errors)
- R3 (`gan config` commands the user invokes)
